* 1. 概要
本アプリは、React Native と Firebase を組み合わせたクイズ形式の学習・暗算アプリである。
小学校1年生レベルから大学レベルまでの難易度を選択し、全20問のクイズに回答する。
スコア（正解数・経過時間）に応じて、アーケードゲームのように上位20名がランキングに名前を登録できる仕組みを持つ。
最小の運用コスト（ほぼ無料枠内）での運用と、チート（不正行為）の完全な防止を両立する設計とする。

* 2. アーキテクチャと量産戦略
** 2.1 テクノロジー構成
- フロントエンド: React Native (TypeScript)
- バックエンド/サーバー: Firebase (Cloud Functions, Firebase Storage, Firestore)

** 2.2 量産化戦略 (Flavor / Target の採用)
ソースコード（プログラム）は完全に1つに共通化し、Androidの「Flavor」およびiOSの「Target」機能を採用する。
これにより、以下の専門ジャンルアプリを設定の切り替えだけで一括ビルド・量産可能とする。
- 「暗算クイズアプリ」
- 「歴史年号クイズアプリ」
- 「英単語クイズアプリ」
- 「漢字・元素記号クイズアプリ」

* 3. 画面仕様とゲームループ
** 3.1 画面フロー
1. *難易度選択画面*: 小1〜大学レベルを選択。スタートボタンでクイズ開始。
2. *クイズ画面*: 1問ごとにユーザーが「回答」ボタンで回答を送信する。問題文は LaTeX 形式で配信され、
   =MathDisplay= コンポーネントが WebView を使わずネイティブで数式として表示する。
   回答ボタン押下後、入力表示エリアに正誤の ○/× がアニメーション演出付きで重なって表示される。
   20問すべてに回答すると結果画面へ自動遷移する。
3. *結果画面*: =submitScoreFunction= のレスポンスをもとに正解数・経過時間を表示する。
   ランクイン（=ranked: true=）の場合は上位20位分のランキングテーブルを合わせて表示する。
   ランキングのレスポンスが20件未満の場合、不足行はすべて「—」で埋めて常に20行表示する。
   自分のランク行は =accent= カラーでハイライト表示する。
   （名前入力フォームは今後実装予定。）
4. *ランキング画面*: タブバー右側（トロフィーアイコン）のタブで常時アクセス可能。
   =getRankingFunction= を呼び出してレベルごとのランキングを表示する。
   画面上部の横スクロールタブ（A〜M の13レベル）でレベルを切り替える（デフォルトは M = 小学1年生）。
   切り替え時に選択中レベルの学年名・難易度星を表示する。
   認証不要のため、未プレイのユーザーでも閲覧できる。
   自分のランク強調なし（=RankingTable= コンポーネントを =myRank= 指定なしで使用）。

* 4. フロントエンド入力UI仕様および制御
ユーザーの入力ブレや不正な文字列生成を未然に防ぐため、OS標準のキーボードは使用せず、記述式クイズ（有理数範囲）に対応した以下の専用カスタムUIおよびバリデーションを実装する。

** 4.1 搭載コンポーネント・ボタン
- =+/-= 切り替えボタン（トグル式の符号フラグ制御）
- =0= 〜 =9= の数値ボタン
- =/= （スラッシュ）ボタン
- =削除= ボタン
- =回答= ボタン（最終問題では「回答して結果へ」ラベルに変化する。押下中は =isSubmittingRef= で再入を防止）

** 4.2 入力バリデーション制御（ボタンの活性・非活性）
- *スラッシュ制限*: 文字列内にすでに =/= が存在する場合、2回目の =/= 入力は受け付けない（ボタンを非活性化）。
- *先頭スラッシュ禁止*: 入力文字数が0の状態で =/= の入力は受け付けない。
- *分母のゼロ禁止*: =/= 入力直後に =0= を入力することは受け付けない（分母が0になる状態の防止）。

** 4.3 画面上の注意書き文言（UI明記ルール）
理不尽な不正解判定によるユーザーの離脱を防ぐため、クイズ画面の入力エリア周辺に以下のルールを常時明記する。
#+BEGIN_QUOTE
【解答ルール】
・分数は「1/3」や「-2/5」のようにスラッシュを用いて入力してください。
・回答は必ずこれ以上約分できない状態（既約分数）にしてください。
・計算結果が整数になる場合は、分母をつけずに整数（例: 3）で入力してください（「3/1」は不正解となります）。
・「-0」は不正解となります。
#+END_QUOTE

** 4.4 正誤オーバーレイ表示仕様
回答ボタン押下後、入力表示エリア（=NumericKeypad= 内の =display= View）の上に
○/× を重ねてアニメーション表示する。

- *正解*: 赤い ○（=#E02020=）をバウンス（=withSpring=）アニメーションで表示
- *不正解*: グレーの ×（=#555555=）を同様のアニメーションで表示
- *次問または結果画面への遷移*: 表示から 900ms 後に自動で次問へ進む（または結果画面へ遷移）
- *実装方式*: =NumericKeypad= コンポーネントが =resultState?: boolean | null= prop を受け取り、
  =display= View の内部に =position: absolute= の =Animated.View= で重ねて描画する。
  =resultState= が =null= のときは条件付きレンダリングでアンマウントし確実に非表示にする。
- *タッチ透過*: オーバーレイは =pointerEvents="none"= を設定し、下のボタン操作を妨げない。

| =resultState= 値 | 表示内容         |
|------------------+------------------|
| =true=           | 赤い ○           |
| =false=          | グレーの ×       |
| =null= / 未指定  | 非表示（アンマウント） |

* 4.5 結果画面仕様
** 4.5.1 表示内容
=submitScoreFunction= のレスポンスをもとに以下を表示する。

| 表示項目     | レスポンスフィールド | 表示形式              | 表示条件                  |
|--------------+----------------------+-----------------------+---------------------------|
| 正解数       | =correct_count=      | =N / 20=              | 常時                      |
| 経過時間     | =elapsed_time=       | =XX.X秒=（小数第1位） | 常時                      |
| ランキング   | =rankings=           | 20行固定テーブル      | =ranked: true= のときのみ |

ランキングテーブルの列構成: 順位 / ユーザー名 / 正解数 / 経過時間

- レスポンスの =rankings= が20件未満の場合、不足行はすべて各セルを「—」で埋めて常に20行表示する。
- 自分のランク行（=rank= フィールドが一致する行）を =accent= カラーでハイライト表示する。

ランキングテーブルは =features/quiz/components/RankingTable.tsx= として独立したコンポーネントに切り出されており、

// ランキング画面での使用（ハイライトなし）
<RankingTable rankings={rankings} colors={colors} />
#+END_SRC

** 4.5.2 二重送信防止（ランキング重複登録の防止）
クイズ終了後に =submitScoreFunction= が複数回呼ばれると、同一プレイヤーが1回の回答で
複数回ランク登録できてしまうため、以下の2段階の対策を実施する。

| 対策                     | 実装箇所                  | 防止対象                                                             |
|--------------------------+---------------------------+----------------------------------------------------------------------|
| =gestureEnabled: false=  | =app/_layout.tsx=         | iOS スワイプバックによる =/result= → =/quiz= への戻り                |
| =hasNavigatedToResultRef= | =app/quiz.tsx=           | スワイプバック後または再押下による二重 =router.push=                 |
| =AppButton= スロットル   | =components/AppButton.tsx= | アプリ全体のボタン連打（500ms 以内の再タップをスロットルで無効化）  |
| =isSubmittingRef=        | =app/quiz.tsx=            | 回答処理中（crypto 非同期 + 900ms 待機中）の回答ボタン再押下         |

=hasNavigatedToResultRef= は =useRef(false)= で管理する。
=setTimeout= コールバック内で =isLastQuestion= のとき、フラグが =true= なら即 =return=、
=false= なら =true= に立ててから =router.push= することで、二重遷移を完全に防止する。

*=AppButton= による連打防止*:
アプリ全体の =Pressable= / =TouchableOpacity= を =components/AppButton.tsx= に統一した。
=AppButton= は =onPress= に 500ms のスロットルを内蔵し、インターバル内の再タップを自動的に無視する。
=throttleMs= prop で間隔を変更でき、数値キーパッド（=NumericKeypad=）では =throttleMs={100}= を指定している。

*=isSubmittingRef= による回答ボタン再入防止*:
回答ボタンは押下後に =Crypto.digestStringAsync=（非同期）と =setTimeout= 900ms が直列で走る。
=AppButton= の 500ms スロットルが切れた後も処理が継続しているため、
=isSubmittingRef = useRef(false)= でガードし、=handleSubmit= 先頭で =true= にセット、
=setTimeout= コールバック内（次問遷移または結果画面遷移の直前）で =false= にリセットする。

*バックグラウンド→フォアグラウンド復帰について*:
アプリのバックグラウンド/フォアグラウンド切り替えではコンポーネントはアンマウントされないため、
=useEffect(fn, [])= は再実行されない。この経路では重複登録は発生しない。

* 5. 正誤判定・ローカルハッシュ処理仕様
テンポの良さを重視し、1問ごとにサーバーを叩く通信は行わない。記述式クイズにおける正誤判定は、サーバー負荷軽減およびチート対策との整合性を保つため、例外を認めない「厳格な一致」を原則とする。

** 5.1 解答の数学的制限
- *既約分数の義務化*: 分数で解答する場合、必ずこれ以上約分できない「既約分数」の状態での入力を必須とする。途中の約分忘れ（例: =2/4=）は、計算の過程が合致していてもシステム上「不正解（即バツ）」として処理する。
- *整数の分数表記禁止*: 計算結果が整数（例: =3=）になる場合、分数形式（例: =3/1=）での入力は「約分未実施」とみなし、すべて「不正解（即バツ）」とする。
- *符号「-0」の禁止*: 数値 =0= に対して負の符号を付与した =-0= は、数学的定義および標準表記のルールに反するため「不正解（即バツ）」とする。

** 5.2 ローカル判定とハッシュ化の流れ
- ユーザーが回答ボタンを押した際、アプリ側での自動約分（GCD計算）や符号の救済ロジックは一切行わない。
- UIの内部状態から単純結合された文字列（例: =2/4=, =3/1=, =-0=）のまま =expo-crypto= の
  =Crypto.digestStringAsync(CryptoDigestAlgorithm.SHA256, inputRaw)= で直接 SHA-256 ハッシュ化を実行し、
  サーバーから取得した =answer_hash= と突合して、その場でマルバツの演出を出す。
- サーバー側（Firebase StorageのJSON）には、完全にクレンジングされた既約状態の正解文字列（例: ="1/2"=, ="3"=, ="0"=）から生成されたハッシュ値のみを格納しておく。
- ハッシュ比較は完全にクライアントサイドで完結し、1問ごとのサーバー通信は発生しない。
  全20問の回答後、問題 ID と生の解答のペア =[ { id, answer }, ... ]= 形式でまとめてサーバーへ送信し、
  Functions 側で厳密に一から再検証する（§7.1 参照）。

* 6. データ管理と配信設計
** 6.1 問題データの配置
問題データは複数箇所で管理せず、サーバー側（Firebase Storage）にジャンル・難易度ごとのJSONファイルとして一元管理する。アプリ側には問題データを直接同梱せず、起動時またはクイズ開始時に取得する。

** 6.2 JSONデータの構造（チート対策：サーバー側ハッシュ化）
スマホメモリからの答えの盗み見を防ぐため、クライアントには生の正解を渡さず「ハッシュ値（暗号）」のみを渡す。
Storage 側の JSON には生の正解（=answer=）を保持し、Cloud Functions がクライアントに返す際にサーバー側で SHA-256 ハッシュ化する。これにより、Storage JSON の管理が容易になり、問題作成時にハッシュ値を事前計算する手間が不要となる。

- *Storage 側 JSON（管理用・非公開）*:
#+BEGIN_SRC json
{
  "id": "a001",
  "question": "5 + 7 = ?",
  "answer": "12"
}
#+END_SRC

- *クライアントへのレスポンス（Cloud Functions 経由）*:
#+BEGIN_SRC json
{
  "order": 1,
  "id": "a001",
  "question": "5 + 7 = ?",
  "answer_hash": "ed076287532e82e1d668..."
}
#+END_SRC

=question= フィールドは *LaTeX 形式の文字列* で格納される（例: =\frac{-b \pm \sqrt{b^2-4ac}}{2a}=）。
クライアントは =MathDisplay= コンポーネントに渡すことで数式として描画する（デリミタの付与は不要）。

** 6.3 サーバー側でのキャッシュ管理（コスト削減）
Cloud Functions のインスタンス永続性を利用し、一度 Storage からダウンロードしたJSONファイルを Functions 側のメモリにキャッシュ（一時保存）する。これにより、ユーザー数が増加しても Storage の通信量をほぼゼロ（無料枠内）に抑え込む。
- *問題更新時のキャッシュクリア手法*: JSONファイルのURLにバージョン（例: =_v2.json=）を含め、Functions のデプロイ時に数値を切り替えることで古いキャッシュを強制破棄する。もしくはキャッシュに「有効期限（TTL）」を設定する。

* 7. 鉄壁のセキュリティ・チート対策
** 7.1 想定されるチート手口への防御
- *手口1：関数のフッキング（True/False の書き換え）*
  - *影響*: アプリ画面上はすべて「正解」の演出に書き換えられる。
  - *防御*: 最終スコアは自己申告データではなく、ユーザーが入力した「生の解答リスト」をサーバーに送り、Functions 側で厳密に一から答え合わせを行うため、ランキング登録は100%阻止される。
- *手口2：メモリのスキャン（答えの自動割り出し・マクロ自動入力）*
  - *防御*: スマホメモリ内には暗号（ハッシュ値）しか存在しないため、スクリプトによる答えの自動検出は不可能。

** 7.2 通信データの改ざん・タイマー不正の防御
クイズ終了時に「解答リスト」と「経過時間」を Cloud Functions に送信。Functions 側で「クイズ開始時のタイムスタンプ」と「終了時のタイムスタンプ」の差分をサーバー側で引き算し、リアルな経過時間を算出。人間には不可能な異常タイム（例: 0.1秒など）や、解答リストの不正はサーバー側で破棄する。

* 8. データベース設計 (Firebase Firestore)
ランキング情報は、同時書き込み時のデータ整合性を Firestore Transaction で保証するため、
難易度ごとに *1ドキュメント* で管理し、上位20件のスコアを配列フィールドとして保持する構造とする。
サブコレクションを使わないことで、Transaction 内の読み書きを最小（1 read / 1 write）に抑え、
無料枠内でのコスト最適化も実現する。

** 8.1 コレクション構造
#+BEGIN_EXAMPLE
/rankings (コレクション)
    └ /level_A (ドキュメント: 難易度ごとに 1 つ。level_A 〜 level_M の 13 ドキュメント)
          scores: [                          ← 上位 20 件を常にソート済み配列で保持
            {
              username:      "AAAAA",       (文字列: 5文字・英大文字。登録前は "-----")
              correct_count: 20,            (数値: 正解数)
              elapsed_time:  45.2,          (数値: 経過時間・秒)
              created_at:    2026-06-13T09:27:00Z  (タイムスタンプ)
              claim_token:   "uuid-v4",     (文字列: 一時トークン。username 登録後に削除)
            },
            ...                             (最大 20 件)
          ]
#+END_EXAMPLE

** 8.2 ランキング更新ロジック（Cloud Function 内 Transaction）
Firestore Transaction を使用し、読み取り→判定→書き込みをアトミックに実行する。
複合インデックスは不要。ソートはサーバー側（Cloud Function）で行う。

#+BEGIN_EXAMPLE
runTransaction(db, async (t) => {
  1. /rankings/level_{level} を 1 read
  2. scores 配列が 20 件未満
       OR 新スコアが現在の 20 位より上位（※）
     → 配列に新エントリを追加
  3. 新エントリは username: "-----"、claim_token: UUID v4 で初期化
  4. correct_count 降順 → elapsed_time 昇順でソート
  5. 先頭 20 件に切り捨て（20件超は末尾を削除）
  6. ドキュメントを 1 write で上書き
})
// 同時書き込み競合時は Firestore が自動リトライ → 整合性保証
// claim_token は registerUsernameFunction 呼び出し後に削除される（TTL: 10分）
#+END_EXAMPLE

*** ランクイン判定条件（※）
- =correct_count= が 20 位の値より *大きい*、または
- =correct_count= が同値 かつ =elapsed_time= が 20 位の値より *小さい*

*** 同率タイの扱い
20 位と完全同スコア・同タイムの場合、=created_at= の早い方（先着）を上位とする。
新エントリを追加後にソートするため、同率では末尾（新エントリ）が自然に切り捨てられる。

* 9. Cloud Functions API 仕様
** 9.1 getRankingFunction（ランキング一覧取得）

指定したレベルの最新ランキング上位20件を返す。ユーザー認証不要。

*** 9.1.1 呼び出し方法

#+BEGIN_SRC typescript
const getRanking = httpsCallable(functions, "getRankingFunction");
const result = await getRanking({ level: "A" });
#+END_SRC

クライアント側ラッパー: =features/quiz/api/getRanking.ts= の =getRanking(level: string)= を使用する。

*** 9.1.2 リクエスト

| フィールド | 型       | 必須 | 説明                           |
|------------+----------+------+--------------------------------|
| =level=    | =string= | ○    | 取得するレベルの識別子（A〜M） |

*** 9.1.3 レスポンス

#+BEGIN_SRC json
{
  "rankings": [
    { "rank": 1, "username": "BBBBB", "correct_count": 20, "elapsed_time": 30.1 },
    { "rank": 2, "username": "CCCCC", "correct_count": 20, "elapsed_time": 35.2 },
    { "rank": 3, "username": "-----", "correct_count": 18, "elapsed_time": 52.4 }
  ]
}
#+END_SRC

| フィールド      | 型               | 説明                                                          |
|-----------------+------------------+---------------------------------------------------------------|
| =rank=          | =number=         | 順位（1〜20）                                                 |
| =username=      | =string=         | ユーザー名。未登録エントリは ="-----"= と表示される           |
| =correct_count= | =number=         | 正解数                                                        |
| =elapsed_time=  | =number=         | 経過時間（秒）                                                |

ランキングデータが存在しない場合（誰もプレイしていない）は =rankings: []= を返す（エラーにしない）。
=claim_token= はレスポンスに含まれない。

* 10. 運用・コスト管理
** 10.1 コスト見込み
通常運用（月間数万プレイ規模）であれば、Cloud Functions・Firestore・Storage の無料枠内にほぼ収まるため、固定費（Artifact Registry の保管料数十円/月）のみで運用可能。

** 10.2 安全対策
開発中およびリリース後の事故を防ぐため、Firebaseに「予算アラート（例: 500円超過で通知）」を必ず設定し、バグによる無限ループ等の課金スパイクを防止する。
