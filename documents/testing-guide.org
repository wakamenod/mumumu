* 1. テスト戦略

** 1.1 テストピラミッド

テスタビリティを最大化するため、以下の3層構造を採用する。

#+BEGIN_SRC text
        ▲ Integration Tests  （少数・画面をまたぐ操作フローを検証）
       ▲▲▲ Component Tests   （中程度・ユーザー操作と振る舞いを検証）
    ▲▲▲▲▲▲▲ Unit Tests       （多数・高速・純粋なドメインロジック）
#+END_SRC

| 層          | 対象                          | ツール                        |
|-------------+-------------------------------+-------------------------------|
| Unit        | types.ts / constants.ts など  | Jest（React 不要・超高速）    |
| Component   | LevelStepper など             | @testing-library/react-native |
| Integration | 画面遷移・router.push の検証   | @testing-library/react-native |

** 1.2 テスタビリティ原則

1. *実装詳細をテストしない* — アニメーションの内部状態（=translateX.value= など）は検証しない。「ユーザーが見て操作できること」を軸にする。
2. *=getByRole= / =getByLabelText= ファースト* — アクセシビリティ属性で要素を特定し、DOM 構造変更に強いテストにする。=getByTestId= は最終手段。
3. *アニメーションはモック* — Reanimated の =setUpTests()= でモック化し、アニメーション完了を待たずに振る舞いをテストする。
4. *ロジックをアニメーションから分離* — 状態更新（=onIndexChange= 等）をアニメーションコールバックの外で先に呼ぶことで、テスト環境でも確実に動作させる。
5. *CI に統合* — =just check= にテストを組み込み、リグレッションを自動検知する。

** 2.2 RTL v14 の重要な変更点

=@testing-library/react-native= v14 では =render()= および =fireEvent= の全メソッドが *async* に変更された。
内部で =await act(...)= を呼ぶため、テストコードでは以下の規約を厳守すること。

*** =render()= は必ず =await= する

#+BEGIN_SRC typescript
// ✅ 正しい
await render(<MyComponent />);

// ❌ 誤り — screen が初期化されず「render function has not been called」エラーになる
render(<MyComponent />);
#+END_SRC

*** =render()= を =act()= でラップしない

=render()= は内部で =act()= を呼ぶため、外側から =act()= でラップすると二重になり
=overlapping act() calls= 警告が発生する。

#+BEGIN_SRC typescript
// ✅ 正しい
await render(<MyComponent />);

// ❌ 誤り — overlapping act() 警告が発生する
await act(async () => {
  render(<MyComponent />);
});
#+END_SRC

*** =fireEvent= は必ず =await= する

=fireEvent.press()=, =fireEvent.changeText()=, =fireEvent.scroll()= 等すべてが async である。
=await= を忘れると =overlapping act() calls= 警告が発生する。

#+BEGIN_SRC typescript
// ✅ 正しい
await fireEvent.press(screen.getByLabelText('回答する'));

// ❌ 誤り — 前の act() と重複する
fireEvent.press(screen.getByLabelText('回答する'));
#+END_SRC

複数の =fireEvent= を連続で呼ぶ場合は、それぞれを =await= する。

#+BEGIN_SRC typescript
// ✅ 正しい — 1つずつ await して直列実行
await fireEvent.press(screen.getByLabelText('A'));
await fireEvent.press(screen.getByLabelText('B'));
await fireEvent.press(screen.getByLabelText('C'));

// ❌ 誤り — 並列に act() が走り overlapping 警告
fireEvent.press(screen.getByLabelText('A'));
fireEvent.press(screen.getByLabelText('B'));
fireEvent.press(screen.getByLabelText('C'));
#+END_SRC

*** =act()= を直接使うケース

RTL v14 においても =act()= を明示的に使う正当なケースが残る。

#+BEGIN_SRC typescript
// タイマーの進行
await act(async () => {
  jest.advanceTimersByTime(1000);
});
#+END_SRC

各テストファイルで再定義する必要はない。

| モック対象                        | 概要                                               |
|-----------------------------------+----------------------------------------------------|
| =react-native-reanimated=        | =setUpTests()= でアニメーションを同期化            |
| =expo-router=                    | =router.push= / =useRouter= / =useLocalSearchParams= を spy 化 |

| =react-native-safe-area-context= | =SafeAreaView= → =View= ラッパーに差し替え         |
| =@react-native-firebase/*=       | =app= / =functions= / =app-check= をスタブ化       |

** 4.2 Firebase callable モック（ファイル単位）

Firebase Cloud Functions を呼び出すテストでは、以下のパターンで =callable= を取得し制御する。
=jest.mock= はファイル最上部に記述し、hoisting により import より先に評価される。

#+BEGIN_SRC typescript
// ─── ファイル最上部 ───
jest.mock('@/lib/firebase', () => {
  const callable = jest.fn();
  return {
    functions: {
      httpsCallable: jest.fn(() => callable),
      useEmulator: jest.fn(),
    },
    firebase: {},
    __callable: callable, // テストからアクセスするためのバックドア
  };
});

/* eslint-disable import/first */
import { render, screen, ... } from '@testing-library/react-native';
import ResultScreen from '@/app/result';
/* eslint-enable import/first */

// ─── callable の取得 ───
// eslint-disable-next-line @typescript-eslint/no-require-imports
const firebaseMock = require('@/lib/firebase');
const callable: jest.Mock = firebaseMock.__callable;

// ─── 各テストでの使い方 ───
beforeEach(() => {
  callable.mockClear();
  callable.mockResolvedValue({ data: MOCK_RESPONSE });
});
#+END_SRC

*** 複数の Cloud Functions を呼び出す画面のモック

=result.tsx= のように =submitScoreFunction= と =registerUsernameFunction= を両方呼ぶ場合、
関数名に応じて異なる callable を返すモックを使用する。

#+BEGIN_SRC typescript
jest.mock('@/lib/firebase', () => {
  const submitCallable = jest.fn();
  const registerCallable = jest.fn();
  return {
    functions: {
      httpsCallable: jest.fn((name: string) => {
        if (name === 'registerUsernameFunction') return registerCallable;
        return submitCallable;
      }),
      useEmulator: jest.fn(),
    },
    firebase: {},
    __callable: submitCallable,
    __registerCallable: registerCallable,
  };
});

const firebaseMock = require('@/lib/firebase');
const callable: jest.Mock = firebaseMock.__callable;             // submitScore 用
const registerCallable: jest.Mock = firebaseMock.__registerCallable; // registerUsername 用
#+END_SRC

すべてのモックは =jest.mock()= によるインラインモック、

** 6.2 ローディング状態のテスト（AbortController パターン）

永遠に resolve しない Promise を設定し、ローディング中の UI を検証する。
テスト終了時に =abort()= で Promise を reject し、リソースリークを防ぐ。

#+BEGIN_SRC typescript
let loadingAbortController: AbortController | null = null;

function setupLoadingResponse() {
  loadingAbortController = new AbortController();
  const { signal } = loadingAbortController;
  callable.mockReturnValue(
    new Promise<never>((_, reject) => {
      signal.addEventListener('abort', () => reject(new Error('aborted')));
    })
  );
}

afterEach(async () => {
  // cleanup() で先に unmount → useEffect のクリーンアップ（cancelled = true）を実行。
  // これにより abort() 後の reject → catch でも setSubmitState は呼ばれない。
  await cleanup();
  loadingAbortController?.abort();
  loadingAbortController = null;
});
#+END_SRC

*** =cleanup()= と =abort()= の順序が重要

=abort()= は Promise を reject し、catch ブロック内の =setState= が呼ばれる可能性がある。
コンポーネントが unmount 済みであれば =useEffect= のクリーンアップで =cancelled = true= が設定されるため
=setState= は実行されない。したがって *必ず =cleanup()= を先に呼ぶ* こと。

#+BEGIN_SRC text
正しい順序:  cleanup() → abort()   （unmount 後なので setState は呼ばれない）
誤った順序:  abort() → cleanup()   （unmount 前なので setState が act() 外で実行され console.error）
#+END_SRC

** 6.3 Fake Timers（タイマー制御）

=setTimeout= を使うフロー（正誤フィードバックの表示時間など）では
=jest.useFakeTimers()= で時間を制御する。

#+BEGIN_SRC typescript
jest.useFakeTimers();

// ... テスト操作 ...

await act(async () => {
  jest.advanceTimersByTime(1000); // 1秒後のフィードバック消去を発火
});

expect(screen.queryByText('⭕')).toBeNull();

jest.useRealTimers();
#+END_SRC

* 11. よくある落とし穴と対策

** 11.1 =overlapping act() calls=

| 原因                                                   | 対策                                          |
|--------------------------------------------------------+-----------------------------------------------|
| =render()= を =act()= でラップしている                 | =await render()= のみにする                   |
| =fireEvent.press()= を =await= していない              | =await fireEvent.press(...)= にする           |
| 複数の =fireEvent= を同期的に連続呼び出ししている       | 各呼び出しを =await= で直列化する             |

** 11.2 =not wrapped in act(...)=

| 原因                                                         | 対策                                                     |
|--------------------------------------------------------------+----------------------------------------------------------|
| =afterEach= で unmount 前に =abort()= → state 更新が発生     | =cleanup()= を先に呼んでから =abort()= する              |
| =jest.advanceTimersByTime()= を =act()= 外で呼んでいる       | =await act(async () => { jest.advanceTimersByTime(...) })= |

** 11.3 =render function has not been called=

| 原因                                          | 対策                            |
|-----------------------------------------------+---------------------------------|
| =render()= を =await= していない（RTL v14）    | =await render(...)= にする     |

** 11.4 テストのタイムアウト・ハング

| 原因                                                    | 対策                                                                       |
|---------------------------------------------------------+----------------------------------------------------------------------------|
| =afterEach= の =act()= 内で pending Promise を待ち続ける | =cleanup()= で先に unmount し、=act()= を使わずに =abort()= する           |
| ローディングテスト後に Promise がリークしている           | =AbortController= パターンで明示的に reject する                           |
