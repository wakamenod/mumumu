* 1. テスト戦略

** 1.1 テストピラミッド
| 層          | 対象                          | ツール                        |
|-------------+-------------------------------+-------------------------------|
| Unit        | types.ts / constants.ts       | Jest（React 不要）            |
| Component   | 個別コンポーネントの振る舞い  | @testing-library/react-native |
| Integration | 画面単位の操作フロー          | @testing-library/react-native |

** 1.2 原則
1. *実装詳細をテストしない* — アニメーション内部状態は検証しない。ユーザーが見て操作できることを軸にする
2. *アクセシビリティクエリ優先* — =getByLabelText= > =getByText= > =getByRole= > =getByTestId=（最終手段）
3. *アニメーションはモック* — Reanimated は =jest.setup.ts= でモック済み
4. *ロジックをアニメーションから分離* — 状態更新をアニメーションコールバックの外で先に呼ぶ

* 2. RTL v14 の必須ルール

=@testing-library/react-native= v14 では =render()= と =fireEvent= が全て *async* になった。
以下を厳守しないとテストが壊れる。

** 2.1 =render()= は必ず =await= する
#+BEGIN_SRC typescript
await render(<MyComponent />);  // ✅
render(<MyComponent />);        // ❌ 「render function has not been called」エラー
#+END_SRC

** 2.2 =render()= を =act()= でラップしない
=render()= は内部で =act()= を呼ぶため、外側の =act()= は二重になる。
#+BEGIN_SRC typescript
await render(<MyComponent />);                    // ✅
await act(async () => { render(<MyComponent />); }); // ❌ overlapping act() 警告
#+END_SRC

** 2.3 =fireEvent= は必ず =await= する（各呼び出しを直列に）
#+BEGIN_SRC typescript
await fireEvent.press(button1);  // ✅ 1つずつ await
await fireEvent.press(button2);
#+END_SRC

** 2.4 =act()= を直接使う正当なケース
#+BEGIN_SRC typescript
await act(async () => { jest.advanceTimersByTime(1000); });  // タイマー進行
#+END_SRC

* 3. テストファイルの規約

** 3.1 配置
=features/{name}/__tests__/= に配置する。共通コンポーネントは =components/__tests__/=。

** 3.2 ファイルヘッダー
各テストファイルの先頭にテスト方針を記載した JSDoc コメントを置く。

** 3.3 =describe= / =it= の構造
- 最上位 =describe= はコンポーネント名・画面名
- 2階層目は機能的な関心事でグループ化
- =it= の説明文は *日本語* で、検証内容を具体的に記述する

** 3.4 テストデータ
モックレスポンスはファイル上部に定数として定義。命名: =MOCK_RESPONSE_= + 状態名。

* 4. モックパターン

** 4.1 グローバルモック
=jest.setup.ts= で定義済み。各テストファイルで再定義不要。
対象: Reanimated, expo-router, expo-font, SafeAreaContext, Google Mobile Ads, Firebase 等。
→ *=jest.setup.ts= の実ファイルを参照せよ*

** 4.2 Firebase callable モック
=jest.mock('@/lib/firebase', ...)= でファイル最上部に定義し、
=__callable= バックドアでテストからアクセスする。

- 単一 callable: =__callable= を export
- 複数 callable（例: submitScore + registerUsername）: =httpsCallable= 内で関数名分岐し、=__callable= / =__registerCallable= 等を export

→ *既存テストファイル（=ResultScreen.test.tsx= 等）を参照せよ*

** 4.3 =__mocks__/= ディレクトリは使用しない
全て =jest.mock()= インラインまたは =jest.setup.ts= で完結させる。

* 5. 非同期テストのパターン

** 5.1 =waitFor= で状態遷移を待機
API 呼び出し後の UI 更新は =waitFor= で待つ。

** 5.2 ローディング状態テスト（AbortController パターン）
永遠に resolve しない Promise + =AbortController= でローディング UI を検証する。

重要: *=cleanup()= を先に呼んでから =abort()=* する（順序逆だと =not wrapped in act= エラー）。

** 5.3 Fake Timers
=setTimeout= を使うフローでは =jest.useFakeTimers()= で制御。
=afterEach= で =jest.useRealTimers()= に復元すること。

* 6. rAF コンポーネントのテスト

=requestAnimationFrame= を使うコンポーネント（ElapsedTimer 等）の注意点:

- Jest 29 の fake timers は rAF もフェイク化する
- =jest.advanceTimersByTime(16)= で1フレーム分発火
- =Date.now()= 制御: =jest.setSystemTime()= を使う。ただし =advanceTimersByTime(16)= が +16ms するためオフセットに注意
- =AppState= モック: =jest.spyOn(AppState, 'addEventListener')= でリスナーを取得し手動発火

→ *=ElapsedTimer.test.tsx= を参照せよ*

* 7. よくある落とし穴

| 症状                              | 原因                                     | 対策                                      |
|-----------------------------------+------------------------------------------+-------------------------------------------|
| =overlapping act() calls=        | =render()= を =act()= で囲んだ          | =await render()= のみにする              |
| =overlapping act() calls=        | =fireEvent= を =await= していない        | =await fireEvent.press(...)= にする      |
| =not wrapped in act(...)=        | =abort()= → =cleanup()= の順で呼んだ    | =cleanup()= → =abort()= の順にする      |
| =not wrapped in act(...)=        | =advanceTimersByTime= を =act()= 外で実行 | =await act(async () => { ... })= で囲む |
| =render function has not been called= | =render()= を =await= していない    | =await render(...)= にする              |
| テストのタイムアウト・ハング      | pending Promise のリーク                 | AbortController パターンで明示的に reject |
